# Plan 039 — Schema-driven settings sync (hybrid)

## Problem

Every time the bridge adds a setting, the Mac has to mirror it. The drift
is real and concrete:

| Cycle | Bridge added | Mac drift |
|---|---|---|
| Plan 035-037 | 5 image sliders + presets + datestamp toggle + 5 format presets | Mac model + UI never updated for months |
| 2026-05-30 (commit `9c5e240`) | We just shipped the catch-up: 10-field `BridgeAdjustmentsConfig` + sliders + toggles + preset picker | Took a dedicated executor pass to find + close all the gaps |

Per new setting, the cost is roughly:
- Bridge config dataclass field + serialise/parse code
- Bridge LCD row + render branch + help text + i18n
- Mac `BridgeConfig.swift` field + CodingKeys + defaults
- Mac `BridgeSettingsDraft.swift` validation rule + diff emit
- Mac `BridgeSettingsView.swift` SwiftUI row + binding
- Tests on both sides + L() strings on both sides

Six file changes across two repos for one knob. And the Mac side
silently drifts the moment we forget.

## Decision (user-confirmed 2026-05-30)

**Hybrid (Option C):** schema-driven for knobs pages, hand-written for
product-context pages.

| Surface | Approach | Why |
|---|---|---|
| **Adjustments** | Schema-driven (Phase 1) | Pure knobs page — sliders / toggles / pickers. Generic rendering suffices. |
| Personalisation | Schema-driven (future) | Same — appearance / font / language pickers. |
| Printer | Hand-written | Model-specific copy, pairing flow, status integration. |
| Network | Hand-written | FTP wizard, hotspot vs same-Wi-Fi decision, masked secrets. |
| Updates | Hand-written | Preflight + multi-phase progress + rollback flow. |
| Backup | Hand-written | Confirmation flow + Keychain handling + restore-to-different-bridge. |
| Diagnostics | Hand-written | Log streaming + support-bundle copy + recovery flow. |

Schema-driven only where the section IS the data. Hand-written where the
section needs prose, wizards, or product polish that doesn't generalize.

## Phase 1 scope (this work)

Adjustments only. Closes the largest drift surface and validates the
architecture without touching anything that already works well.

### A. Bridge: `GET /v1/config/schema/adjustments` endpoint

Returns a JSON document describing the adjustments section. Generated by
introspecting the existing `AdjustmentsConfig` dataclass (one-time
introspection helper, then auto-updates as new fields are added to the
dataclass).

Shape:

```json

  "schema_version": 1,
  "section": "adjustments",
  "title": "Image adjustments",
  "fields": [
    {
      "key": "preset",
      "type": "picker",
      "label": "Preset",
      "help": "Choose a preset or Custom slot",
      "options": [
        {"value": "Default", "label": "Default"},
        {"value": "Vivid", "label": "Vivid"},
        {"value": "Soft", "label": "Soft"},
        {"value": "Black & white", "label": "Black & white"},
        {"value": "Instax Film", "label": "Instax Film"},
        {"value": "Custom1", "label": "Custom1"},
        ...
      ]
    },
    {
      "key": "saturation",
      "type": "slider",
      "label": "Saturation",
      "help": "Colour intensity. Negative dulls, positive boosts",
      "range": {"min": -100, "max": 100, "step": 1},
      "display": "signed_percent"
    },
    {"key": "exposure",  "type": "slider", "label": "Exposure",  ...},
    {"key": "sharpness", "type": "slider", "label": "Sharpness", ...},

    },
    {
      "key": "datestamp",
      "type": "toggle",

    },
```

The strings (`label`, `help`) are the English source per project convention
(plan 037 polish established plain-English-as-key). The Mac's L() helper
re-uses them as lookup keys, falling back to English when unmapped. Future
translation work is shared.

Bridge-side implementation:
- New `bridge/src/instantlink_bridge/manager/schema.py` module with a
  `build_adjustments_schema()` function that returns the dict above.
  Reads enum members + dataclass field defaults to enumerate options
  programmatically.
- Wire `GET /v1/config/schema/adjustments` into `manager/api.py`. Signed
  request required (consistent with `/v1/config`).
- Tests: schema includes all 10 fields; preset options include the 5
  built-ins + 6 Custom slots; datestamp format options match the 5
  `DatestampFormat` enum members; depends_on relationships are
  declared.

### B. Mac: schema model + generic renderer

#### B1. `macos/InstantLink/Core/BridgeConfigSchema.swift` (new file)

Decode the schema JSON into typed Swift structs:

```swift
struct BridgeConfigSchema: Codable, Equatable, Sendable {
    let schemaVersion: Int
    let section: String
    let title: String
    let fields: [BridgeConfigField]
}

enum BridgeConfigField: Codable, Equatable, Sendable {
    case picker(BridgePickerField)
    case slider(BridgeSliderField)
    case toggle(BridgeToggleField)
    case text(BridgeTextField)
    case unknown(key: String, type: String)  // forward-compat for future field types
}

struct BridgePickerField: Codable, ... {
    let key: String
    let label: String
    let help: String?
    let options: [BridgePickerOption]
    let dependsOn: BridgeFieldDependency?
}

struct BridgeSliderField: Codable, ... {
    let key: String
    let label: String
    let help: String?
    let range: BridgeFieldRange
    let display: BridgeSliderDisplay  // .signedPercent | .unsignedPercent | .integer
}

// etc.
```

The `case unknown(key:type:)` is critical: if the bridge ships a new field
type (e.g. `"color_picker"`) before the Mac understands it, the Mac
renders a placeholder "Unsupported field — update InstantLink" instead of
crashing.

#### B2. `Core/BridgeTransport.swift` + `Core/BridgeHTTPTransport.swift`

Add `func getAdjustmentsSchema(device:) async throws -> BridgeConfigSchema` to
the protocol + HTTP implementation + InMemory mock.

#### B3. `Features/Bridge/BridgeSchemaSectionView.swift` (new file)

Generic SwiftUI section that takes a `BridgeConfigSchema` and a binding
into a typed draft, renders the fields. Per type:

- `picker`: SwiftUI `Picker` bound to the draft's adjustments enum/string
  field via a key→KeyPath adapter.
- `slider`: SwiftUI `Slider(value:in:step:)` with a trailing value badge.
  `display: .signedPercent` formats as `+50` / `0` / `-25`;
  `.unsignedPercent` as `40`.
- `toggle`: SwiftUI `Toggle`.
- `text`: SwiftUI `TextField`.
- `unknown`: dimmed row "Setting requires a newer InstantLink. Update
  the app to manage <field.key>."

Dependency gating: each field reads its `dependsOn` and applies
`.disabled(!parentValue == expectedValue)` so format pickers gate behind
toggles correctly without the Mac needing per-field gating logic.

#### B4. `Features/Bridge/BridgeSettingsView.swift`

Replace the existing hand-written `adjustmentsCard` body with a single
line:

```swift
BridgeSchemaSectionView(
    schema: draft.schemas[.adjustments],
    draft: draft
)
```

The hand-written card structure (BridgeSettingsSection wrapper, errorFooter
binding) stays — only the field rendering becomes schema-driven.

#### B5. `Core/BridgeSettingsDraft.swift`

- Add `schemas: [BridgeSchemaSectionKind: BridgeConfigSchema]` cache.
- Load the adjustments schema on draft `load()` (parallel to fetching
  `/v1/config`).
- Add `getValue(forKey:) -> Any?` and `setValue(_:, forKey:)` accessors
  that map the bridge's snake_case keys into the typed
  `BridgeAdjustmentsConfig` fields. This is the small adapter layer that
  bridges schema-driven rendering to typed Swift storage.
- Validation reads ranges from the schema — `validate()` walks each
  slider's `range` and rejects out-of-bounds. Bridge owns the source of
  truth for constraints.

### C. Migration path

The typed `BridgeAdjustmentsConfig` stays. The schema-driven UI binds to
it through the new key adapter. Existing tests around typed access still
pass; new tests cover the schema endpoint + the rendering loop.

When a new field is added to the bridge dataclass:
1. Bridge dataclass adds the field. Schema endpoint auto-reflects it.
2. Mac fetches the new schema. The new field renders generically.
3. Mac model adds a typed field IF a future iteration wants typed access
   for that specific field. Until then, the value flows through the
   draft's key-adapter as `Any` and round-trips fine — the bridge sees
   any value the Mac sends back.

Optional: the draft's `BridgeAdjustmentsConfig` could store a
`extraFields: [String: BridgeJSONValue]` map for unknown fields, so the
typed struct doesn't have to be updated immediately for round-trip to
work. Defer until needed.

## Out of scope (Phase 1)

- Migrating Personalisation or any other section to schema-driven.
- Removing the typed `BridgeAdjustmentsConfig` (it stays as the storage
  layer behind the schema renderer).
- Localization workflow changes (the schema strings serve as L() keys
  for now; future polish can introduce explicit string IDs if needed).
- Schema versioning beyond `schema_version: 1` (single-version + Mac
  graceful-degrade on `.unknown` is enough for Phase 1).
- A schema for `BridgeConfig` as a whole — Phase 1 ships
  `/v1/config/schema/adjustments` only.

## Future phases

- Phase 2 — Personalisation section schema + renderer.
- Phase 3 — Audit the remaining hand-written cards: anything that's
  purely a knobs page (idle poweroff?) can migrate.
- Phase 4 — Drop the typed `BridgeAdjustmentsConfig` field-by-field if
  the round-trip through schema becomes the canonical path.
